# Natively v2.8 Release Analysis: "Intelligence OS" Architectural Transition

This document provides a comprehensive release audit of the Natively v2.8 development cycle, evaluating changes introduced from the baseline v2.7.0 commit (`be7280b`) to the current `HEAD` (`33feffb`).

v2.8 marks a major evolution of Natively, transitioning from a stateless audio-to-chat assistant into a stateful, context-aware "Intelligence OS." Key features include Hindsight long-term memory, time-aware SessionMemory, deterministic answer humanization, local code verification, and a regionalized STT relay architecture.

## 1. New Features Added

### Hindsight Long-Term Memory (LTM)
* **Description:** An optional, user-provisioned long-term memory sidecar service that runs locally or in the cloud. It indexes past meeting transcript segments, custom profiles, and documents, making them queryable via semantically matching vector search.
* **Implementation:** Orchestrated via `HindsightManager.ts` (`electron/services/HindsightManager.ts`).
* **Graceful Degradation:** If Hindsight is unconfigured or unreachable, it degrades gracefully to a "Noop" memory state. Local meeting memory and search are used as fallbacks without throwing main-process errors.
* **Developer helpers:** Added python sidecar startup and configuration scripts:
  * `scripts/hindsight-dev-server.py`
  * `scripts/hindsight-start.sh`

### Live Session Memory & Entity Resolver
* **Description:** A time-decayed session memory engine that processes rolling transcript turns and resolves long-range demonstrative follow-ups (e.g. "Tell me more about *that project*" -> resolves to "Natively" based on prior conversation context).
* **Files:**
  * `electron/llm/SessionMemory.ts`
  * `electron/llm/liveSessionMemory.ts`
  * `electron/llm/sessionFollowupResolver.ts`
  * `electron/llm/transcriptEntityExtractor.ts`
* **Mechanics:** Extracts entities (projects, skills, company, people, topics, decisions, compensation) deterministically from speaker roles and cues. It uses half-life recency decay (measured strictly in seconds to match the `SessionTracker`) to calculate entity salience.

### Style-Only Spoken Answer Humanizer
* **Description:** A deterministic style-only post-processor that rewrites LLM raw draft answers to make them sound natural for spoken conversation.
* **Files:** `electron/llm/humanLikeness.ts`
* **Mechanics:**
  * Deterministically strips out corporate/LinkedIn buzzwords ("unique blend", "leverage", "streamline", "cutting-edge").
  * Strips third-person references to the user ("the candidate") and rewrites them in the first person ("I").
  * Bans structural artifacts unfit for voice output (e.g. em/en dashes, empty bullets).
  * Preserves critical inline HTML bold formatting (`**bold**`).

### Answer Diversity Guard & Polish
* **Description:** Prevents repetitive speech outputs across a single long meeting session.
* **Files:** `electron/llm/answerPolish.ts`
* **Mechanics:**
  * Computes Jaccard similarity and sentence-structure matching against recent answers.
  * Compresses boilerplate chat scaffolding ("Based on the screen context...", "Sure, here is...") into direct, speakable prose.

### Local Sandboxed Code Verification
* **Description:** Automatically extracts code blocks from LLM answers and verifies their logical correctness by executing them inside local sandboxed processes against test cases, before displaying a "Verified ✓" badge.
* **Files:** `electron/llm/codeVerification/`
* **Mechanics:**
  * Spawns isolated processes (`node` or `python3`) with bounded resource constraints: 3s SIGKILL timeout, 256KB output cap, scrubbed environments, closed `stdin`, and concurrency semaphore.
  * Local SQL verification using `sqlite3 -safe -bail :memory:` with a Dialect Firewall that skips dialect-specific errors instead of failing.
  * One-shot correction call: If code fails, it performs a single correction attempt to fix the bug in place, streaming the corrected code to the UI.

### Natively Browser Extension Companion
* **Description:** A Chrome/Browser companion extension that captures the DOM context of the active tab, extracts source code and text contents, and pipes it directly to the desktop app's intelligence context.
* **Files:** Located in the `natively-browser/` directory, includingpopup interface, service worker logic, esbuild configurations, and README/privacy contracts.

## 2. Existing Features Improved

### Speech-to-Text (STT) Provider Pipelines
* **Description:** Reconfigured pipelines for Deepgram, Whisper, and Soniox STT integrations. Added robust pre-population of masked API keys.
* **UI Error Mapping:** Improved handling of trial expiration and billing exhaustion errors (e.g., Soniox `trial_expired` maps to a clear download-to-local action instead of crashing or showing raw JSON errors).

### Screen OCR & Vision Processing
* **Description:** Scoped the cropping tool (`lazy Cropper`) directly to its active window to prevent coordinate bleed across multi-monitor setups.
* **Registry Order:** Reorganized `VisionProviderRegistry` to optimize local vs cloud image-processing tasks and enforce platform-specific order.

### Skills & Modes Managers
* **Modes Cache:** Added hybrid transcript chunk caching (`ModeHybridRetriever`) to reduce duplicate database lookups during vector indexing.
* **macOS Ventura Guard:** Added compatibility checks to process monitor taps, preventing permissions crashes on newer macOS versions.

## 4. Performance Improvements

* **Mode-Hybrid Retriever Chunk Cache:** Reduces RAG latency by caching text chunks, preventing duplicate index lookups.
* **Rolling Transcript Token Cap:** Limits transcript size to keep prompt sizes bounded over long sessions, reducing LLM prefill costs.
* **Physics-Spring Window Resize:** Replaced expensive CSS `clip-path` and JavaScript interval animation logic with physics-spring animations (`overlayResizeEasing.mjs`), ensuring buttery smooth resizing of the overlay.

## 5. Architecture Changes

### Transition to "Intelligence OS" Control Plane
* Changed the LLM integration from a stateless single-turn prompt generator to a stateful control plane. Active session modes (e.g. Sales, Interview, Lecture) are treated as *priors* for the routing intent classifier, improving intent classification accuracy.

### Regional STT Relay Migration
* Migrated the realtime STT egress pipeline off the centralized Railway control plane. Traffic is now routed directly to regional VPS instances (e.g., `us-relay.natively.software` and `asia-relay.natively.software`). The Railway server is preserved solely as a failover control plane.

### Sidecar Subprocess Lifecycle
* Implemented detached process group lifecycle hooks for local subprocesses (Hindsight database, Ollama, Codex CLI). Process groups are killed via POSIX `SIGKILL` on shutdown to prevent orphaned sidecar processes.

## 7. Telemetry, Logging, Analytics & Observability

### Structured PII-Free Telemetry (`piTelemetry.ts`)
* **Markers-Only Design:** Logs non-sensitive indicators only (event names, recalled entity KINDS, age buckets, error classifications, latency times). It never records raw transcript texts, custom context, resumes, or credentials.
* **Double-Layer Privacy Guard:**
  1. **Allowlist:** Only variables declared in `ALLOWED_KEYS` are processed; all others are dropped.
  2. **Value Scrubbing:** Strings are limited to 48 characters. A regex filter (`SENSITIVE_VALUE_RE`) automatically flags and drops values matching salary, social security, or credit card patterns.
* **Age Bucketing:** Replaces exact epoch timestamps with coarse intervals (`immediate`, `1-5min`, `5-15min`, etc.).

### Remote Analytics Sinks
* Integrated preflight health checks and error sinks for Axiom, PostHog, and Sentry in production builds, separated from local debug consoles.

## 9. Profile Intelligence, Memory & Context Routing

### Mode-Aware Priors & Context Routing
* The active session mode acts as a safety boundary for memory access:
  * **Coding Mode:** Limits memory retrieval strictly to technical topics, blocking resume and project history to prevent context pollution.
  * **Sales Mode:** Restricts queries to company dossiers, competitor objections, and personal profiles.
  * **Negotiation Mode:** The *only* mode permitted to retrieve compensation and salary data.

### Value-Level Compensation Guard
* If any extracted entity value matches `SALARY_VALUE_RE` (e.g. "120k", "$150,000/yr"), it is automatically forced into the `comp` kind, preventing it from being accidentally recalled outside of `negotiation` mode.

## 10. Security, Privacy & Permissions

* **Local Database Encryption:** Integrated SQLCipher file-level encryption for the sqlite storage file, binding keys to the macOS Keychain or Windows Credentials Manager.
* **Cross-Mode Leak Boundaries:** Prevents internal leakage of sensitive personal files (resumes, interview preparation notes) into public developer sessions.
* **Zero PII Telemetry:** Scrubbing blocks raw values from being emitted to third-party endpoints.

## 11. Backend, Server & API Changes

### Supabase Quota Leases (`migrations/004_stt_quota_lease.sql`)
* Implemented a transaction-scoped advisory lock (`pg_advisory_xact_lock`) on the hash of the user identity. This serializes concurrent `stt_reserve_session` calls and prevents a double-billing TOCTOU bug where overlapping requests could bypass remaining quota ceilings.

### Control Plane Reaper Loop
* A server-side reaper daemon (`STT_REAPER_INTERVAL_MS`) sweeps the `stt_sessions` database table, automatically marking sessions older than the active timeout as `abandoned` and releasing their reserved quota.

### Health Writes
* Health monitors write status events directly to the `relay_health_events` table for region-specific latency reporting.

## 12. Electron, Main Process & IPC Changes

## 13. Release Rollout & Operational Checklist

This checklist outlines the sequential rollout steps required to deploy the v2.8 "Intelligence OS" release safely.

### Phase 1: Database Migration (Supabase Backend)
1. Apply `003_deploy_regional_relays.sql` schema updates (health tables, reaper procedures).
2. Apply `004_stt_quota_lease.sql` schema updates (advisory lock procedures, quota reservations, triggers).
   > [!IMPORTANT]
   > Ensure migrations are applied *before* updating the control plane server to prevent database mismatch errors on startup.

### Phase 2: Regional Relay Provisioning
1. Deploy STT relay containers to regional VPS hosts:
   * US region: `us-relay.natively.software`
   * Asia region: `asia-relay.natively.software`
2. Verify `/healthz` endpoints are accessible and returning `200 OK`.
3. Verify TLS certificates are valid and match client-side hardcoded paths.

### Phase 3: Control Plane Environment Configuration
Set the following environment variables on the primary Fastify control plane:
* `STT_QUOTA_LEASE_ENABLED=true` — Activates transaction-scoped advisory lock reservation.
* `STT_REAPER_INTERVAL_MS=60000` — Sweeps abandoned sessions every 60 seconds.
* `STT_REAPER_ABANDON_AFTER_SECONDS=1800` — Reclaims sessions inactive for 30 minutes.
* `STT_RELAY_ALLOW_CLIENT_LATENCY_PROBES=true` — Allows client geo-latency metrics to override default routing.
* `STT_RELAY_ENABLE_PERCENT=25` — **CRITICAL:** Restrict regional relay routing to a 25% canary group during initial rollout.

### Phase 4: Hindsight Setup & Configuration
1. For users utilizing Hindsight long-term memory:
   * Verify the local python vector database server starts cleanly using `scripts/hindsight-start.sh`.
   * Configure the target endpoints and models within the client's new "Intelligence" settings tab.

### Phase 5: Client-Side E2E Verification
